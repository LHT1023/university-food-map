## 🎯 产品背景与痛点分析

### 场景

某高校联盟（3-5 所大学）周边聚集了大量餐饮商户，但学生在"吃什么"这个高频决策上普遍面临困境。

### 核心痛点

| 痛点 | 描述 | 现有方案的不足 |
|------|------|---------------|
| **信息分散** | 餐厅信息散落在美团、大众点评、小红书、朋友圈等多个平台 | 没有一个聚焦"大学城"场景的统一入口 |
| **决策疲劳** | 每天 3 次"吃什么"的选择，学生普遍出现"选择困难症" | 外卖平台推荐算法偏向商业化，不贴近学生真实需求 |
| **缺乏学生视角** | 学生关心的维度（性价比、出餐速度、深夜营业、聚餐适合度）在通用平台上不突出 | 现有评价体系没有针对学生场景的筛选维度 |
| **场景化需求强** | "穷鬼模式""考完试犒劳""宿舍聚餐""深夜复习后宵夜"等场景有强烈的个性化需求 | 现有平台不支持按"心情+预算+场景"组合筛选 |

## 🔍 核心功能详解

### 1. 地图探索

- **SVG 手绘地图**：5 所虚构大学（华光/黎明/晨曦/星河/天远）+ 20 家餐厅的空间分布
- **三维筛选**：菜系（15 种）× 学校（5 所）× 预算滑块（¥0-100+），可叠加使用
- **实时搜索**：输入关键词即时过滤，匹配餐厅名、菜系、标签、描述、学校名
- **空状态处理**：筛选无结果时展示友好引导 + 一键清除筛选按钮
- **详情面板**：从右侧滑入，不遮挡地图，展示评分/价格/距离/评价数/营业时间

### 2. AI 智能推荐（核心差异化功能）

支持**自然语言输入**，用户像跟朋友聊天一样描述需求：


输入："又饿又穷，20块以内吃饱"
引擎解析：预算≤¥20 · 心情:hungry
输出：兰州正宗牛肉面(¥15) / 沙县小吃(¥12) / 蜜雪冰城(¥8)
每家附带个性化推荐理由


详见下方 [NLU 引擎设计](#-nlu-智能推荐引擎技术亮点)。

### 3. 今天吃什么

- Canvas 绘制的转盘动画，包含 10 家餐厅
- 物理缓动曲线（cubic-bezier）模拟真实转盘减速
- 结果可直接收藏

### 4. 学生评价

- 预置 8 条不同风格的真实感评价
- 支持用户发布：星级评分 + 文字 + 标签（性价比高/味道好/量大/深夜好去处 等 10 个标签）
- 按餐厅筛选、按点赞数排序

### 5. 收藏系统

- **全局统一**：在地图、AI 推荐卡片、转盘结果、详情面板中均可一键收藏
- **实时同步**：任意入口的收藏操作会立即同步到所有 UI（地图标记变红心、侧栏心形变化、收藏页更新、AI 卡片按钮状态变化）
- **持久化**：基于 localStorage，刷新不丢失

## 🧠 NLU 智能推荐引擎（技术亮点）

这是本项目的核心技术亮点——一个**纯前端的自然语言理解 + 推荐引擎**，无需任何后端或 API 调用。

### 架构


用户自然语言输入
       ↓
┌─────────────────┐
│  NLU.parse()    │  意图解析层
│  正则关键词匹配    │  提取 5 个维度
└────────┬────────┘
         ↓
   ┌─────┴─────┐
   │ budgetMax  │  预算上限（如 ≤¥20）
   │ budgetMin  │  预算下限（如 ≥¥40）
   │ cuisines[] │  口味偏好 → 菜系映射
   │ scene      │  用餐场景（聚餐/约会/宵夜/快速/solo/早餐）
   │ mood       │  当前心情（累了/开心/饿了/伤心/纠结）
   └─────┬─────┘
         ↓
┌─────────────────┐
│  NLU.score()    │  权重评分层
│  对 20 家餐厅     │  逐一计算匹配分
│  加权打分          │
└────────┬────────┘
         ↓
   评分规则：
   ├── 基础分 = rating × 5
   ├── 预算匹配：+30（在预算内）/ -40（超预算）
   ├── 菜系匹配：+35
   ├── 场景匹配：+25~30
   ├── 心情匹配：+10~20
   ├── 评价数加成：+3/条
   └── 收藏加成：+5
         ↓
┌─────────────────┐
│  NLU.greet()    │  回复生成层
│  个性化问候语      │  基于 mood/scene 选择模板
│  + Top 3 推荐     │  每家附带 ≤2 条匹配理由
└─────────────────┘


### 意图解析示例

| 用户输入 | 解析结果 |
|---------|---------|
| "又饿又穷，20块管饱" | budget≤20, mood=hungry |
| "带对象吃饭，有氛围的" | scene=date |
| "宿舍4人聚餐，量大就行" | scene=group, people=4 |
| "夜猫子求宵夜推荐" | scene=late |
| "减脂期有没有低卡的" | cuisines=[素食,粤菜,面食] |
| "考完试想犒劳自己！" | mood=tired, budget≥40 |

### 为什么不用 LLM API？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 调用 LLM API | 理解能力更强 | 需要 API Key、有延迟、可能失败、有成本 |
| **本地 NLU 引擎** ✅ | **零延迟、零成本、100%可用、可解释** | 理解能力有限（但对于20家餐厅的场景完全够用） |

在"20 家餐厅 × 有限场景"的问题规模下，基于关键词匹配的 NLU 引擎是**更优的工程选择**——更快、更稳、更可控。同时通过展示"🧠 理解: 预算≤¥20 · 心情: hungry"的调试信息，让用户看到引擎的解析过程，增强信任感。

## 💡 产品设计决策

### 为什么用地图而不是列表？

大学城是一个**空间概念**——学生选择餐厅时，"离我的教学楼/宿舍多远"是核心因素。地图视图让空间关系一目了然，比纯列表更贴合真实决策场景。

### 为什么做"今天吃什么"转盘？

数据显示大量用户搜索"随便""都行""不知道"——他们不是没有选择，而是**不想承担选择的认知负担**。转盘把决策权交给"命运"，用游戏化的方式解决选择疲劳。

### 为什么详情面板从侧边滑出而不是弹窗？

弹窗（Modal）会打断用户的空间认知——用户在地图上建立的"哪家店在哪"的心理模型被遮挡了。侧边滑出面板让地图保持可见，用户可以同时看到餐厅位置和详细信息。

### 为什么收藏要全局同步？

用户可能在 AI 推荐中发现一家店 → 收藏 → 切到地图想看位置 → 地图上应该立刻显示红心标记。如果收藏状态不同步，用户会困惑"我刚收藏的店去哪了"。这是**状态一致性**的基本要求。

## 🛠️ 技术选型与实现

| 层面 | 选择 | 理由 |
|------|------|------|
| **框架** | 纯 HTML/CSS/JS（零依赖） | 单文件部署，打开即用，无构建步骤 |
| **地图** | SVG 手绘 + JS 交互 | 无需地图 API Key，完全可控，离线可用 |
| **推荐引擎** | 自研 NLU（正则 + 权重评分） | 零延迟、零成本、100%可用 |
| **数据持久化** | localStorage | 收藏数据刷新不丢失 |
| **动画** | CSS transitions + Canvas | 转盘用 Canvas 绘制，其余纯 CSS 实现 |
| **字体** | DM Sans + Noto Sans SC + ZCOOL KuaiLe | 西文/中文/标题 三套字体分工明确 |
| **响应式** | CSS Grid + Media Queries + 底部 Tab | 桌面/平板/手机 三端适配 |
